# LumiTask v0.2 演进方案

> 目标：保持轻量化定位，增强任务依赖管理、远程 Agent 支持、通知机制、桌面应用
> 原则：通用产品，低门槛，不绑定特定 Agent 平台

## 一、当前架构现状

### 本地依赖关系图

```
LumiTask (Next.js + SQLite)
  ├── Agent Adapter (pluggable)
  │   ├── Claude Code Adapter → spawn 本地 CLI
  │   └── OpenClaw Adapter → spawn 本地 CLI + 读本地 session 文件
  ├── Session Observer → 直接读 ~/.openclaw/agents/*/sessions/*.jsonl
  ├── Cron Detection → execSync("openclaw cron list --json")
  └── Agent Detection → execSync("openclaw agents list --json")
```

**核心问题**：所有 OpenClaw 交互都依赖**本地文件系统 + 本地 CLI 进程**，无法支持远程场景。

### 模块 1：Cron 检测修复（P0 / 0.5 天）

#### 问题

`src/app/api/cron/route.ts` 的 `parseJsonFromOutput()` 有 JSON 解析 bug：
- OpenClaw 返回嵌套对象 `{ "jobs": [{ "schedule": {...} }] }`
- 解析器用 `{`/`}` 深度匹配，但只跟踪一种括号类型
- 遇到内层嵌套 `}` 提前闭合 → 截取到不完整 JSON → 返回空数组

#### 方案

**方案 A：修复 JSON 解析器**（推荐，兼顾 CLI 和本地文件）

```typescript
function parseJsonFromOutput(output: string): unknown {
  const clean = stripAnsi(output);
  // 找到第一个 [ 或 {
  let start = -1;
  for (let i = 0; i < clean.length; i++) {
    if (clean[i] === '[' || clean[i] === '{') { start = i; break; }
  }
  if (start < 0) return [];

  // 用栈匹配所有括号类型
  const stack: string[] = [];
  const pairs: Record<string, string> = { '{': '}', '[': ']' };
  const closers = new Set(Object.values(pairs));
  let end = -1;

  for (let i = start; i < clean.length; i++) {
    const ch = clean[i];
    if (pairs[ch]) {
      stack.push(pairs[ch]);
    } else if (closers.has(ch)) {
      if (stack.length === 0) break;
      if (stack[stack.length - 1] !== ch) break;
      stack.pop();
      if (stack.length === 0) { end = i; break; }
    }
  }

  if (end < 0) return [];
  try { return JSON.parse(clean.slice(start, end + 1)); }
  catch { return []; }
}
```

**方案 B：增加本地文件回退**

当 CLI 解析失败时，直接读取 `~/.openclaw/cron/jobs.json`：

```typescript
function readLocalCronJobs(): CronJob[] {
  const jobsPath = join(OPENCLAW_HOME, 'cron', 'jobs.json');
  if (!existsSync(jobsPath)) return [];
  const data = JSON.parse(readFileSync(jobsPath, 'utf-8'));
  return (data.jobs || []).map(mapOpenClawJob);
}
```

**建议**：A + B 同时实施。A 修复根因，B 作为 fallback。

### 模块 2：任务依赖与结构化 I/O（P0 / 2 天）

#### 目标

让有上下文依赖的任务可以被拆解和串联：
- 任务 A 的输出自动作为任务 B 的输入
- 任务 B 等待任务 A 完成后才执行
- 支持并发控制，避免资源冲突

#### 2.1 Schema 扩展

```sql
-- tasks 表新增字段
ALTER TABLE tasks ADD COLUMN depends_on TEXT;        -- JSON string[]，前置任务 ID
ALTER TABLE tasks ADD COLUMN input_context TEXT;      -- JSON，结构化输入
ALTER TABLE tasks ADD COLUMN output_result TEXT;      -- JSON，结构化输出
ALTER TABLE tasks ADD COLUMN concurrency_key TEXT;    -- 互斥锁 key
ALTER TABLE tasks ADD COLUMN retry_count INTEGER DEFAULT 0;
ALTER TABLE tasks ADD COLUMN max_retries INTEGER DEFAULT 0;
ALTER TABLE tasks ADD COLUMN parent_task_id TEXT REFERENCES tasks(id); -- 父任务（用于拆解）
```

设计考量：
- `dependsOn` 存 JSON 数组，轻量且灵活，不需要额外关联表
- `parentTaskId` 支持任务拆解（大任务拆成子任务），与 `dependsOn` 独立
- `maxRetries` 默认 0（不重试），用户/Agent 按需设置
- `concurrencyKey` 可选，只有需要互斥的场景才设值

#### 2.2 并发锁管理器

新建 `src/lib/agents/concurrency.ts`：

```typescript
// 内存锁，进程重启自动释放（符合轻量化定位）
const locks = new Map<string, string>(); // key → taskId

export function canAcquire(key: string, taskId: string): boolean {
  const holder = locks.get(key);
  return !holder || holder === taskId;
}

export function acquire(key: string, taskId: string): boolean {
  if (!canAcquire(key, taskId)) return false;
  locks.set(key, taskId);
  return true;
}

export function release(key: string, taskId: string): void {
  if (locks.get(key) === taskId) locks.delete(key);
}
```

#### 2.3 调度器增强

`src/lib/agents/task-scheduler.ts` 的 `tick()` 增加：

```
对于每个待执行任务：
  1. 检查 dependsOn 中所有前置任务是否 status=done
     - 如有前置任务 failed → 当前任务也标记 failed（级联失败）
     - 如有前置任务未完成 → 跳过，等下一轮
  2. 检查 concurrencyKey 是否可获取
  3. 检查 scheduleAt 是否到时间
  4. 全部通过 → 获取锁 → 执行
```

#### 2.4 执行器增强

`src/lib/agents/task-executor.ts` 改动：

```
执行前：
  - 解析 inputContext，注入到 Agent 的 prompt 中
  - 如果有 dependsOn，收集前置任务的 outputResult 合并到上下文

执行后（成功）：
  - 将 Agent 返回的结构化结果存入 outputResult
  - 释放 concurrencyKey 锁

执行后（失败）：
  - retryCount < maxRetries → 重新入队（status=open）
  - retryCount >= maxRetries → 标记 failed
  - 释放 concurrencyKey 锁
```

#### 3.3 通知发送机制

```typescript
// 通知发送不直接调飞书/Discord API
// 而是让 OpenClaw 代为投递（通过已有频道基础设施）

async function sendNotification(payload: NotificationPayload): Promise<void> {
  const { agentId, channel, accountId } = await getNotificationConfig();

  // 方式 1：通过 OpenClaw CLI 发送消息
  // openclaw message send --channel feishu --account default --text "任务 #7 已完成"
  execAsync(`openclaw message send --channel ${channel} --account ${accountId} --text "${formatMessage(payload)}"`);

  // 方式 2（远程模式）：通过 Gateway RPC
  // rpc('message.send', { channel, accountId, text: formatMessage(payload) })
}

function formatMessage(payload: NotificationPayload): string {
  // 根据事件类型格式化
  switch (payload.event) {
    case 'task.completed':
      return `✅ 任务 #${payload.taskNumber} 已完成\n${payload.title}\n${payload.actionUrl}`;
    case 'task.failed':
      return `❌ 任务 #${payload.taskNumber} 失败\n${payload.title}\n原因：${payload.body}`;
    case 'task.blocked':
      return `⚠️ 任务 #${payload.taskNumber} 需要你的决定\n${payload.title}\n${payload.body}\n${payload.actionUrl}`;
  }
}
```

#### 3.4 触发时机

| 事件 | 级别 | 说明 | 备注 |
|------|------|------|------|
| `task.completed` | info | 任务完成 | 可选通知 |
| `task.failed` | error | 任务失败（含失败原因） | 默认通知 |
| `task.blocked` | warning | 任务阻塞，需人工介入 | 默认通知 |
| `task.dependencies_met` | info | 依赖就绪，自动执行中 | 仅通知，任务已自动开始执行 |

> **关于 `task.dependencies_met`**：如果用户创建任务时设置了 `dependsOn`，当所有前置任务完成后，调度器**自动执行**该任务，不需要用户再操作。这个通知只是告知"你的任务链正在推进"。

#### 4.5 Remote Client（新增）

```typescript
class RemoteOpenClawClient implements OpenClawClient {
  private gatewayUrl: string;
  private token: string;

  constructor(url: string, token: string) {
    this.gatewayUrl = url;
    this.token = token;
  }

  // 所有操作通过 Gateway RPC
  async listAgents() { return this.rpc('agents.list'); }
  async executeAgent(agentId, message) { return this.rpc('agent.run', { agentId, message }); }
  async listCronJobs() { return this.rpc('cron.list'); }
  async listChannels() { return this.rpc('channels.list'); }
  async sendMessage(channel, accountId, text) {
    return this.rpc('message.send', { channel, accountId, text });
  }

  private async rpc(method: string, params?: unknown) {
    const httpUrl = this.gatewayUrl.replace('ws://', 'http://').replace('wss://', 'https://');
    // HTTP POST with Bearer token
    const res = await fetch(`${httpUrl}/rpc/${method}`, {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${this.token}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify(params || {}),
    });
    return res.json();
  }
}
```

#### 4.6 长任务自动检测

**问题**：远程 Agent 执行的长任务能否被自动检测到？

##### 本地模式

已实现。通过 `session-observer.ts` 扫描 `~/.openclaw/agents/*/sessions/` 目录发现活跃 session。

##### 远程模式

分两层：

```
层 1（通用 - 任何 Agent）：Heartbeat API
  Agent 主动调 LumiTask API 上报进度
  POST /api/tasks/:id/heartbeat { progress, tokenUsage }
  适用于所有 Agent，不限 OpenClaw

层 2（OpenClaw 专属）：Gateway 状态轮询
  每 30 秒调 gateway RPC: status
  返回各 agent 当前是否有活跃 session
  LumiTask 对比本地任务列表，关联匹配
  → 可发现非 LumiTask 发起的长任务
```

##### 用户不通过 LumiTask 发起的任务能检测到吗？

可以。Gateway `status` RPC 返回所有 agent 的当前 session 状态。
LumiTask 在 Dashboard 上显示一个「活跃 Session」面板：

```
┌─ 活跃 Agent Session ──────────────────────────┐
```

### 模块 5：消息决策系统（轻量版）（P1 / 1.5 天）

#### 设计理念

不做 lumiclaw 那样的重型决策流（campaign → plan → approve），而是做**通用的 Agent 请求人类输入**机制：

```
Agent 执行任务时遇到需要人类决定的事 → 创建决策请求 → 用户通过 UI/通知 回复 → Agent 继续
```

这个能力已经部分存在（`task.blocked` + `blockReason` + `/api/tasks/:id/reply`），但缺少结构化。

#### 5.1 扩展 blocked 机制

```typescript
// 当前 block 只有 reason（纯文本）

interface DecisionRequest {
  type: 'confirm' | 'choose' | 'input' | 'approve';
  question: string;
  options?: { id: string; label: string; description?: string }[];
  defaultOption?: string;
  context?: Record<string, unknown>;  // 辅助决策的上下文
}

// 存入 tasks.block_reason 字段（JSON）
// 用户通过 reply API 回复
```

## 三、远程 Agent 支持详解

### 用户视角：连接远程 Agent 只需三步

```
Step 1: 用户在远程服务器终端运行一条命令
  $ openclaw qr --setup-code-only
  → 输出连接码: eyJ1cmwiOiJ3czovLzE5Mi4xNjguMS4...

  或者直接在飞书/Telegram 对 Agent 说：
  "给我 LumiTask 连接码"
  → Agent 读取本地配置，返回连接码

Step 2: 用户在 LumiTask Settings 粘贴连接码
  → 自动解析出 Gateway URL + Token

Step 3: 点击「连接」
  → LumiTask 自动发现所有 Agent、频道、定时任务
  → 完成
```

**不需要**用户安装任何额外工具、不需要手填 URL/Token、不需要理解 Gateway 概念。

### 场景分析

| 场景 | 描述 | 连接方式 | 可行性 |
|------|------|----------|--------|
| **A. 同一台机器** | LumiTask + Agent 都在本机 | 自动检测，无需配置 | ✅ 当前已支持 |
| **B. 局域网服务器** | Agent 在内网服务器 | 连接码（`openclaw qr`）| ✅ |
| **C. 云服务器** | Agent 在云端 VPS | 连接码 + Tailscale/SSH 隧道 | ✅ |
| **D. 容器化** | LumiTask 在 Docker | Gateway + 端口映射 | ✅ |
| **E. 多台机器** | Agent 分布在不同服务器 | 多个连接码 | 🔶 未来 |

### 连接拓扑

```
场景 A（本地 - 自动）：
  LumiTask ──spawn──→ openclaw CLI ──→ Agent

场景 B/C（远程 - 连接码配对）：
  [用户电脑]                          [远程服务器]
  LumiTask ──HTTP/WS──→ Gateway:18789 ──→ Agent(s)
  （连接码 = base64 { url, bootstrapToken }）

场景 C（需要穿透时）：
  选项 1: openclaw gateway run --tailscale serve  (Tailscale MagicDNS)
  选项 2: ssh -N -L 18789:127.0.0.1:18789 user@remote  (SSH 隧道)
  选项 3: openclaw gateway run --bind lan  (同局域网直连)
```

### 长任务自动检测

**核心问题**：远程 Agent 的长任务能否被自动发现？

**答案：可以。** 分三层实现：

```
层 1 — 通用 Heartbeat（任何 Agent 可用）：
  Agent 主动调 LumiTask API 上报进度
  POST /api/tasks/:id/heartbeat { progress, tokenUsage }
  适用于所有 Agent，不限 OpenClaw

层 2 — OpenClaw 本地扫描（已有）：
  LumiTask 定期扫描 ~/.openclaw/agents/*/sessions/
  发现新 session → 关联到对应任务
  仅本地模式可用

层 3 — OpenClaw Gateway 轮询（远程模式）：
  每 30 秒调 rpc('status')
  返回各 agent 当前 session 状态
  可发现用户不通过 LumiTask 发起的长任务
  Dashboard 上显示「活跃 Session」面板：

  ┌─ 活跃 Agent Session ──────────────────────┐
  │ 🟢 main — 正在执行（已运行 5 分钟）        │
  │    未关联任务  [创建任务关联]               │
  │ 🟢 lumi-tiktok-lead — 执行中（12 分钟）    │
  │    关联任务：#15 分析竞品数据               │
  └────────────────────────────────────────────┘
```

**v0.2 实施**：层 1（通用）+ 层 2（保持现有）+ 层 3（新增远程轮询）

## 六、风险与待确认

| 项目 | 风险 | 缓解 |
|------|------|------|
| Gateway RPC 方法不完整 | 可能缺少 session 流式读取 | 先实现轮询，等 Gateway 升级 |
| Electron + better-sqlite3 | 原生模块编译链复杂 | 完全复用 lumiclaw 验证过的构建流程 |
| 任务依赖循环检测 | A→B→A 会导致死锁 | 创建时做 DAG 环检测 |
| 远程网络稳定性 | WebSocket 断连 | 自动重连 + 心跳 + HTTP fallback |
